Explain the JavaScript event loop. What is the difference between microtasks and macrotasks? What does this code log?Senior📊 diagram 
console.log('1');
setTimeout(() => console.log('2'), 0);
Promise.resolve().then(() => console.log('3'));
queueMicrotask(() => console.log('4'));
console.log('5');
// Output: 1, 5, 3, 4, 2

Model answer: JS is single-threaded. Sync code runs on the call stack. Async APIs (timers, fetch, DOM events) are handled by the host (browser/libuv) and their callbacks are queued. The event loop pulls callbacks in when the stack is empty, with a strict priority:

  • Microtasks (Promise callbacks, queueMicrotask, MutationObserver): the entire microtask queue is drained after every task, before rendering.
  • Macrotasks (setTimeout, setInterval, I/O, UI events): one per loop iteration.

Senior follow-ups to expect: a microtask that queues another microtask can starve rendering (infinite microtask loop blocks paint); setTimeout(fn, 0) has a ~4ms clamp after nesting; requestAnimationFrame runs before paint, not in either queue.

Event loop: microtask queue is fully drained before the next macrotask (and before paint).

Worked example of the full cycle (sync → drain microtasks → 1 macrotask → repeat):

console.log('A');                                   // ① sync

setTimeout(() => {                                  // macrotask #1
  console.log('E');
  Promise.resolve().then(() => console.log('F'));   // microtask queued INSIDE macrotask
}, 0);

setTimeout(() => console.log('G'), 0);              // macrotask #2

Promise.resolve().then(() => {
  console.log('C');
  Promise.resolve().then(() => console.log('D'));   // microtask queued by a microtask
});

console.log('B');                                   // ① sync

// Output: A B C D E F G
  1. Sync runs to completion → A B (timers/promises only get registered).
  2. Stack empty → drain ALL microtasks: C, which queues D — still drained in the same pass.
  3. 🎨 Render opportunity — browser may paint here.
  4. 1 macrotask: timeout #1 → E; then microtasks drain again → F.
  5. Next macrotask: timeout #2 → G.

Proof point: F beats G even though G's timer was registered first — the microtask queue is flushed after every macrotask. Render-blocking contrast:

function microLoop(){ Promise.resolve().then(microLoop); } // page frozen — paint never gets a turn
function macroLoop(){ setTimeout(macroLoop); }             // page fine — render runs between tasks

JS chạy 1 luồng. Ưu tiên: code đồng bộ → toàn bộ microtask (Promise) → 1 macrotask (setTimeout) → lặp lại. Microtask chạy hết trước khi render. Ví dụ trên: F chạy trước G dù timer của G đăng ký trước — vì sau mỗi macrotask, hàng đợi microtask được xả sạch.

What is a closure? Give practical use cases and one classic pitfall.Mid+

Model answer: A closure is a function bundled with its lexical environment — it keeps access to variables of the scope where it was defined, even after that scope has returned.

Practical uses: data privacy / module pattern, memoization & caching, debounce/throttle, factory functions, React hooks (every render captures its own closure).

// Classic pitfall: var shares ONE binding across iterations
for (var i = 0; i < 3; i++) setTimeout(() => console.log(i)); // 3 3 3
for (let i = 0; i < 3; i++) setTimeout(() => console.log(i)); // 0 1 2 (let = new binding per iteration)

Senior angle: closures are a common source of memory leaks (long-lived callback capturing a large object) and of stale state in React (useEffect capturing an old value).

Closure = hàm nhớ được scope nơi nó được tạo. Ứng dụng: debounce, memoize, private state. Bẫy: var trong vòng lặp, stale closure trong React.

Tra cứu property đi từ object → prototype → … → null. class chỉ là syntax sugar trên prototype. Phân biệt __proto__ (link của instance) và prototype (property của constructor).

How does this get bound? How do arrow functions differ?Mid+

Model answer: this is determined at call time, by 4 rules (highest priority first):

  1. new → the newly created object.
  2. Explicit: call/apply/bind → the given object (bind is permanent).
  3. Implicit: obj.method() → the object before the dot.
  4. Default: undefined in strict mode / ESM, globalThis otherwise.

Arrow functions have no own this — they capture it lexically from the enclosing scope at definition time, and it cannot be rebound. That's why arrows are ideal for callbacks but wrong for object methods and cannot be constructors.

const obj = { x: 1, a: () => this.x, b() { return this.x } };
obj.a(); // undefined (lexical this = module/global)
obj.b(); // 1
const f = obj.b; f(); // undefined — lost implicit binding

4 quy tắc: new > bind/call/apply > obj.method() > mặc định. Arrow function không có this riêng — lấy từ scope bao ngoài, không rebind được.

Hoisting, the Temporal Dead Zone, and var vs let vs const.Mid+

Model answer: Declarations are processed before execution. var is hoisted and initialized to undefined (function-scoped). let/const are hoisted but uninitialized — accessing them before the declaration line throws a ReferenceError; that window is the TDZ. Function declarations hoist fully (body included); function expressions / arrows follow the variable's rule.

console.log(a); // undefined (var hoisted)
console.log(b); // ReferenceError (TDZ)
var a = 1; let b = 2;

Nuance worth mentioning: const means the binding is immutable, not the value — objects are still mutable (use Object.freeze or immutability discipline for that).

var hoisted = undefined; let/const hoisted nhưng nằm trong TDZ → ReferenceError. const khoá biến, không khoá object bên trong.

Promises deep-dive: states, chaining, error handling, and Promise.all vs allSettled vs race vs any.Senior

Model answer: A Promise is a state machine: pending → fulfilled | rejected (settled, immutable once settled). .then always returns a new promise; returning a value resolves it, throwing rejects it, returning a promise adopts its state — that's what enables chaining.

CombinatorResolves whenRejects whenUse case
Promise.allall fulfill (array)first rejectionparallel calls, all required
Promise.allSettledalways (status objects)neverbatch jobs; report each result
Promise.racefirst settles (either way)first rejection if firsttimeouts
Promise.anyfirst fulfillmentall reject (AggregateError)fastest healthy mirror

Error handling: .catch(f) ≡ .then(undefined, f); a catch recovers the chain. With async/await, use try/catch; remember await in a loop is sequential — collect promises then Promise.all for parallelism. Unhandled rejections crash Node (v15+) — always attach handlers.

// Parallel, not sequential:
const [user, orders] = await Promise.all([getUser(id), getOrders(id)]);

Promise là state machine 3 trạng thái. all fail nhanh, allSettled không bao giờ reject, race lấy cái xong đầu tiên, any lấy cái thành công đầu tiên. Await trong loop = tuần tự.

ES Modules vs CommonJS — key differences, and how does Node.js handle both?Senior

Model answer:

  • CJS (require/module.exports): synchronous, dynamic (can require conditionally at runtime), exports a copied value snapshot of the object, loaded on demand.
  • ESM (import/export): static — parsed before execution, enabling tree-shaking and top-level await; exports are live read-only bindings; async loading; strict mode by default.

Node decides by "type": "module" in package.json or extensions (.mjs/.cjs). Interop rules: ESM can import CJS (default export = module.exports); CJS cannot require() ESM (must use dynamic import()) — though Node 22+ allows require(esm) when the module has no top-level await. Dual packages use the "exports" field with import/require conditions — relevant when publishing an SDK.

Circular deps behave differently: CJS returns a partially-filled exports object; ESM's live bindings usually resolve later reads correctly but can hit TDZ errors.

CJS: sync, giá trị copy, dynamic. ESM: static → tree-shaking, live binding, có top-level await. Publish SDK nên khai báo "exports" với cả 2 condition import/require.

How does garbage collection work in V8? Name common memory-leak sources in JS apps.Senior

Model answer: V8 uses generational, tracing GC (reachability from roots — not reference counting). The heap is split into a small new space (young objects, fast Scavenge/copying collector) and old space (survivors get promoted; collected by Mark–Sweep–Compact, largely incremental/concurrent — Orinoco — to shorten pauses).

Common leak sources:

  • Unremoved event listeners / subscriptions (esp. on long-lived emitters).
  • Closures capturing large structures inside long-lived callbacks/caches.
  • Ever-growing Maps/arrays used as caches without eviction → use WeakMap/WeakRef or LRU with TTL.
  • Detached DOM nodes still referenced from JS.
  • Forgotten timers (setInterval) and unfinished promises holding scopes.

Detection: Chrome DevTools heap snapshots (compare 3 snapshots, look at retained size / detached nodes), process.memoryUsage(), --inspect for Node.

V8: GC theo reachability, 2 thế hệ (new space Scavenge, old space Mark-Sweep-Compact). Leak phổ biến: listener không remove, cache Map không giới hạn, closure giữ object lớn, detached DOM, setInterval quên clear.

Shallow vs deep copy. Compare spread, JSON round-trip, and structuredClone.Mid+

Model answer: Shallow copy ({...obj}, Object.assign, arr.slice()) copies one level — nested objects are shared references. Deep copy options:

  • JSON.parse(JSON.stringify(x)) — loses Date (→ string), Map/Set, undefined, functions; throws on circular refs and BigInt. Legacy trick only.
  • structuredClone(x) — the modern standard: handles Date, Map, Set, ArrayBuffer, circular references. Cannot clone functions, DOM nodes, class methods (prototype is lost).
  • Library (lodash cloneDeep) or hand-rolled for special class instances.

Senior angle: in React/Redux you rarely deep-copy — you do structural sharing (copy only the changed path) for performance; Immer automates it.

Spread chỉ copy 1 tầng. structuredClone là chuẩn hiện đại (xử lý được Date/Map/Set/circular), nhưng không clone function/prototype. Trong React ưu tiên structural sharing thay vì deep copy.

Explain debounce vs throttle. Implement debounce from scratch.Mid+

Model answer: Both limit call frequency. Debounce waits until events stop for N ms (search-as-you-type, resize end, autosave). Throttle guarantees at most one call every N ms while events continue (scroll position, mousemove, chart pan — relevant for charting UIs).

function debounce(fn, ms) {
  let timer;
  return function (...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), ms);
  };
}

Follow-ups: leading/trailing options; cancel method; why this and args forwarding matter; in React wrap in useMemo/useRef so the timer survives re-renders.

Debounce: chờ ngừng thao tác mới chạy (search input). Throttle: chạy đều mỗi N ms khi thao tác liên tục (scroll, pan chart). Trong React phải giữ qua useRef/useMemo.

type vs interface — differences and when to use each?Mid+

Model answer: Mostly interchangeable for object shapes. Key differences:

  • interface: supports declaration merging (two declarations merge — essential for augmenting libs, e.g. extending Express.Request), extends reads slightly better, errors are often clearer.
  • type: can express unions, intersections, tuples, mapped & conditional types, primitives aliases — things interfaces can't.

Practical rule: public API object shapes → interface (extendable/mergeable, good for SDKs); everything else (unions, utilities, function types) → type. Consistency in the codebase matters more than the choice.

Interface: merge được, dùng cho public API/SDK. Type: mạnh hơn (union, conditional, mapped). Quy ước nhất quán trong codebase quan trọng hơn.

Type narrowing: type guards, discriminated unions, unknown vs any, and never.Senior

Model answer: Narrowing = the compiler refining a union inside control flow via typeof, instanceof, in, equality checks, or custom guards (x is T predicates).

type Event =
  | { type: 'click'; x: number; y: number }
  | { type: 'key';   code: string };

function handle(e: Event) {
  switch (e.type) {           // discriminant
    case 'click': return e.x; // narrowed to click
    case 'key':   return e.code;
    default: { const _exhaustive: never = e; return _exhaustive; } // compile error if a variant is added
  }
}
  • Discriminated unions (shared literal tag) are the idiomatic way to model states — e.g. {status:'loading'} | {status:'success', data} | {status:'error', error} makes impossible states unrepresentable.
  • unknown = type-safe any: you must narrow before use. Use for API responses/catch clauses. any disables checking and is contagious.
  • never = unreachable; used for exhaustiveness checks (above) — a compile-time safety net when new variants appear.

Discriminated union + switch + never exhaustive check = mô hình state chuẩn. unknown buộc phải narrow trước khi dùng (an toàn), any tắt hẳn type-check (tránh).

What happens when you type a URL and press Enter? (end to end)Senior📊 diagram

Model answer — walk the layers:

  1. URL parsing & HSTS check; browser cache lookup.
  2. DNS resolution: browser cache → OS cache → resolver → recursive lookup (root → TLD → authoritative). Result cached by TTL.
  3. TCP handshake (SYN → SYN-ACK → ACK), then TLS handshake (cert validation, key exchange; TLS 1.3 = 1-RTT, HTTP/3/QUIC merges transport+TLS over UDP).
  4. HTTP request/response: headers, compression (br/gzip), possibly 301/302 redirects, CDN edge may answer (cache HIT).
  5. Rendering: HTML parsed to DOM (streaming), CSS → CSSOM, JS may block parsing, then render tree → layout → paint → composite (next question).
  6. Subresources fetched (preload scanner runs ahead), page becomes interactive.
URL → DNS → TCP/TLS → HTTP → parse → render. Mention caching layers at every step to stand out.

Senior signal: mention what can be optimized at each hop — DNS prefetch, preconnect, TLS 1.3/QUIC, CDN, cache headers, streaming HTML.

Trình tự: parse URL → DNS (nhiều tầng cache) → TCP/TLS → HTTP (CDN có thể trả cache) → DOM/CSSOM → render. Điểm cộng: nói cách tối ưu ở từng bước (preconnect, QUIC, CDN…).

Explain the Critical Rendering Path. Which resources block rendering, and what do async/defer do?Senior📊 diagram
CRP: DOM + CSSOM → render tree → layout → paint → composite.

Model answer: The browser can't paint until it has DOM + CSSOM. CSS is render-blocking (all of it, unless media-scoped). Sync JS is parser-blocking, and it also waits for CSSOM (it might read styles). Optimizations:

  • <script defer>: download in parallel, execute in order after parsing, before DOMContentLoaded — the default choice.
  • <script async>: execute as soon as downloaded, order not guaranteed — for independent scripts (analytics).
  • Inline critical CSS, preload key assets, minimize CSS, code-split JS, avoid @import.

CSS chặn render, JS sync chặn parse (và chờ CSSOM). defer = tải song song, chạy đúng thứ tự sau khi parse xong (mặc định nên dùng); async = chạy ngay khi tải xong, không đảm bảo thứ tự.

Reflow (layout) vs repaint vs composite — and how do you minimize them in a chart-heavy UI?Senior

Model answer: Changing geometry (width, top, font-size, DOM insertion) triggers reflow → repaint → composite (most expensive). Changing visuals only (color, background, shadow) triggers repaint → composite. Changing transform/opacity on a promoted layer is composite-only — GPU, cheapest, off the main thread.

  • Avoid layout thrashing: interleaved reads (offsetHeight) and writes force synchronous reflow. Batch: read all, then write all (or use requestAnimationFrame).
  • Animate only transform/opacity; promote sparingly with will-change (layers cost memory).
  • Chart-heavy UIs: prefer canvas (one element, no per-point DOM), batch draws per rAF tick, use contain: layout paint to isolate widgets, virtualize anything DOM-based.
// Thrash (bad): read-write-read-write in a loop
els.forEach(el => { el.style.width = el.offsetWidth + 10 + 'px'; });
// Fix: read phase, then write phase
const w = els.map(el => el.offsetWidth);
els.forEach((el, i) => { el.style.width = w[i] + 10 + 'px'; });

Reflow (đổi geometry) > repaint (đổi màu) > composite (transform/opacity, rẻ nhất). Tránh đọc–ghi xen kẽ layout. Chart nhiều điểm dữ liệu → canvas + gom vẽ theo rAF.

Event propagation: capturing, target, bubbling. What is event delegation and why use it?Mid+📊 diagram
Events travel down (capture), hit the target, then bubble up — delegation listens once at the ancestor.

Model answer: An event travels document → target (capturing, opt-in via {capture:true}), fires at the target, then bubbles back up (default listener phase). stopPropagation() halts the walk; preventDefault() cancels default behavior only.

Delegation: one listener on an ancestor handles events from many children via event.target.closest('li'). Benefits: works for dynamically added rows, one listener instead of thousands (memory), simpler cleanup. This is how React ≤16 worked (events at document) and React 17+ (at root container).

Gotchas: some events don't bubble (focus, blur, mouseenter — use focusin/focusout, mouseover); event.target vs event.currentTarget.

3 pha: capture ↓ → target → bubble ↑. Delegation: 1 listener ở cha xử lý ngàn con (dùng closest()), hoạt động với element thêm động. Lưu ý focus/blur không bubble.

Explain CORS. When does a preflight request happen, and how do credentials work?Senior📊 diagram

Model answer: The Same-Origin Policy blocks JS from reading cross-origin responses (origin = scheme + host + port). CORS is the server's way to opt in, via response headers checked by the browser (curl/servers are unaffected — CORS protects users, not APIs).

  • Simple requests (GET/POST/HEAD with safe headers, form content types): sent directly; browser checks Access-Control-Allow-Origin on the response.
  • Preflighted: anything else (e.g. Content-Type: application/json, Authorization header, PUT/DELETE) → browser first sends OPTIONS with Access-Control-Request-Method/Headers; server must answer with allowed origin/methods/headers (+ Access-Control-Max-Age to cache the result).
  • Credentials (cookies): need credentials:'include' on fetch AND Access-Control-Allow-Credentials: true, and the allowed origin must be explicit — not *.
Preflight flow: OPTIONS asks permission; real request only follows if allowed.

CORS do browser enforce, bảo vệ user. Request "không đơn giản" (JSON, Authorization, PUT/DELETE) → preflight OPTIONS trước. Dùng cookie: credentials:'include' + Allow-Credentials:true + origin cụ thể (không được *).

How does HTTP caching work? Cache-Control, ETag, 304 — and how does a CDN like CloudFront fit in?Senior📊 diagram

Model answer: Two layers:

  • Freshness (no request at all): Cache-Control: max-age=…, s-maxage (shared caches/CDN), immutable, public/private, no-store (never cache) vs no-cache (cache but revalidate).
  • Validation (cheap request): stale entry revalidated with If-None-Match: <ETag> / If-Modified-Since → server answers 304 Not Modified (no body) or 200 with new content.

Production pattern (fits the CV's CDN deployment): hashed asset filenames (app.3f9c.js) + max-age=31536000, immutable; HTML with no-cache so releases propagate instantly. CDN: s-maxage controls edge TTL separately; invalidate on deploy or use stale-while-revalidate for zero-latency refresh.

Each layer answers if fresh; validation (ETag/304) avoids re-downloading bodies.

2 tầng: freshness (max-age, không gửi request) và validation (ETag → 304, request nhẹ). Pattern chuẩn: asset có hash + immutable 1 năm; HTML no-cache. CDN dùng s-maxage riêng.

Core Web Vitals: what are LCP, INP, CLS, and how do you improve each?Senior
  • LCP (Largest Contentful Paint, ≤2.5s): time to render the biggest element. Improve: server/CDN speed (TTFB), preload hero image / fetchpriority="high", inline critical CSS, SSR, compress images (AVIF/WebP).
  • INP (Interaction to Next Paint, ≤200ms — replaced FID in 2024): worst-case interaction latency. Improve: break long tasks (scheduler.yield / chunking), less JS on main thread, web workers for heavy computation (e.g. chart indicator math), avoid huge renders on input — debounce, virtualize, use CSS content-visibility.
  • CLS (Cumulative Layout Shift, ≤0.1): unexpected movement. Improve: width/height (or aspect-ratio) on images/embeds/ad slots, reserve space for dynamic content, font-display: optional/swap + font preloading, never insert banners above existing content.

Senior signal: distinguish lab (Lighthouse) vs field data (CrUX, RUM via web-vitals library) and mention monitoring in production.

LCP ≤2.5s (tối ưu tài nguyên lớn nhất), INP ≤200ms (cắt long task, đưa tính toán vào worker), CLS ≤0.1 (giữ chỗ trước cho ảnh/font/banner). Phân biệt lab vs field data.

Web security: explain XSS and CSRF, and how to prevent each.Senior

XSS — attacker's JS runs in your page's origin (steals tokens, acts as user). Types: stored, reflected, DOM-based. Defenses:

  • Output encoding by default (React escapes JSX text — but dangerouslySetInnerHTML, href="javascript:", and direct innerHTML bypass it). Sanitize rich HTML with DOMPurify.
  • CSP (script-src 'self' 'nonce-…') as defense-in-depth; HttpOnly cookies so tokens can't be read.

CSRF — victim's browser auto-sends cookies with a forged cross-site request (state-changing POST). Defenses:

  • SameSite=Lax/Strict cookies (Lax is default in Chrome — blocks most CSRF).
  • CSRF tokens (synchronizer or double-submit), checking Origin/Referer headers, requiring custom headers (forces CORS preflight).

Memorable contrast: XSS = attacker runs code on your site; CSRF = attacker borrows the user's cookies from another site. Also worth naming: clickjacking (frame-ancestors), dependency/supply-chain attacks, secrets never in frontend bundles.

How does React's reconciliation work? What is Fiber, and why do keys matter?Senior📊 diagram

Model answer: On state change React re-runs the component to produce a new element tree, then diffs it against the previous one with two heuristics: (1) different element type → tear down and rebuild the subtree; (2) lists are matched by key — stable identity lets React move/keep nodes instead of re-creating them. Index-as-key breaks state association when items reorder/insert (wrong input values, lost focus, mangled animations).

Fiber is the reimplementation that makes rendering interruptible: work is split into units (one fiber per component), processed with priorities, pausable/resumable. Render phase (pure, can be discarded/restarted) vs commit phase (DOM mutations, synchronous). This underpins concurrent features (transitions, Suspense).

Stable keys let the diff reuse nodes; index keys force rewrites and corrupt component state.

Diff theo type + key. Key phải ổn định theo identity (id), không dùng index khi list thay đổi thứ tự. Fiber = render có thể ngắt/ưu tiên; render phase có thể huỷ, commit phase sync.

useEffect deep-dive: dependency array, cleanup, and the fetch race-condition bug.Senior

Model answer: Effects synchronize a component with external systems after commit. Deps array = every reactive value read inside (lint enforces this — don't silence it, restructure). Cleanup runs before the next effect and on unmount.

useEffect(() => {
  let cancelled = false;                    // or use AbortController
  const ctrl = new AbortController();
  fetch(`/api/items/${id}`, { signal: ctrl.signal })
    .then(r => r.json())
    .then(data => { if (!cancelled) setItem(data); });
  return () => { cancelled = true; ctrl.abort(); };  // cleanup kills stale request
}, [id]);

Without cleanup, rapid id changes create a race: an older, slower response can arrive last and overwrite newer data. Other senior points:

  • Common mistakes: object/function deps re-created each render (→ infinite loops; fix with useMemo/useCallback or move inside), effects used for derived state (compute during render instead), missing cleanup of listeners/intervals/sockets.
  • Strict Mode double-invokes mount effects in dev precisely to expose missing cleanup.
  • In real apps, prefer React Query/SWR for data fetching — caching, dedup, and cancellation solved.

Effect = đồng bộ với hệ thống ngoài, chạy sau commit. Luôn cleanup (abort fetch, remove listener). Bug race: response cũ về sau đè data mới → cancelled flag/AbortController. Derived state thì tính trong render, đừng dùng effect.

When do you use React.memo, useMemo, useCallback — and when are they harmful?Senior

Model answer: A component re-renders when its parent re-renders (props equality is irrelevant by default) or its own state/context changes.

  • React.memo(C): skip re-render if props are shallow-equal. Only effective if props are referentially stable — which is where the other two come in.
  • useMemo: cache a computed value (expensive derivation, or stabilize an object/array passed to a memoized child).
  • useCallback(fn, deps) ≡ useMemo(() => fn, deps): stabilize function identity for memo'd children / effect deps.

Harmful when: applied everywhere by default — each adds memory + comparison overhead and hides real problems; a single unstable prop (inline object, children JSX) silently defeats memo. Measure first with React DevTools Profiler. Often better fixes: move state down (isolate the changing part), lift content up (pass expensive subtree as children — it doesn't re-render when the wrapper's state changes). React 19's compiler auto-memoizes, making manual memo mostly obsolete going forward.

memo = chặn re-render khi props không đổi (shallow); useMemo/useCallback để giữ reference ổn định. Đừng rải khắp nơi — đo bằng Profiler trước; ưu tiên tách state xuống thấp hoặc đưa subtree vào children.

How do you render 100K+ data points (tables/charts) in React without killing performance?Senior · CV

  • Virtualize lists/tables (react-window / TanStack Virtual): render only visible rows + overscan.

  • Downsample for display (LTTB algorithm, or aggregate to candles/buckets) — the screen only has ~2K horizontal pixels anyway.

DOM không chịu nổi 100K điểm → canvas/WebGL, React chỉ quản lý vỏ. Virtualize bảng. Data streaming: buffer rồi vẽ theo rAF, giữ data trong ref. Downsample (LTTB/candle) vì màn hình chỉ ~2K pixel ngang. Tính nặng → Web Worker.

Explain React 18+ concurrent features: transitions, useDeferredValue, Suspense, automatic batching.Senior
  • Automatic batching: since 18, multiple setState calls are batched into one render everywhere (previously only in event handlers — timeouts/promises caused multiple renders).
  • Transitions: startTransition(() => setFilter(v)) marks an update as non-urgent — React keeps the UI responsive, can interrupt the render if newer urgent input arrives; isPending drives spinners. Classic use: search box (urgent: input text; transition: filtering 10K rows).

Key concept to articulate: concurrency ≠ parallelism — React can pause, resume, or abandon renders so urgent updates jump the queue.

Batching tự động ở mọi nơi từ React 18. Transition = update "không gấp", có thể bị ngắt bởi input khẩn cấp. useDeferredValue = bản trễ của value. Suspense = khai báo loading boundary, nền tảng cho streaming SSR.

How do you decide between local state, Context, Redux/Zustand, and React Query?Senior

  • Server state (data owned by the backend): React Query/SWR — caching, revalidation, dedup, optimistic updates. Storing it in Redux by hand re-implements all of that badly.
  • Local UI state: useState/useReducer, kept as close to usage as possible.
  • Shared client state (theme, auth session, cross-widget chart settings): Context for low-frequency values; Zustand/Redux for frequently-updated or complex state.

Zustand avoids this with selector-based subscriptions. Senior signal: "most apps need far less global state than they think — server state to React Query, and the rest shrinks dramatically."

Controlled vs uncontrolled components — trade-offs at form scale?Mid+

SSR vs CSR vs SSG, and what exactly is hydration? What is a hydration mismatch?Senior

Hydration: the client re-renders the same tree and attaches event listeners to existing server HTML instead of rebuilding it. SSR = HTML từ server + hydration (gắn listener vào HTML sẵn có). Mismatch xảy ra khi server render khác client (Date, random, window) → flicker. Nội bộ/dashboard sau login thường không cần SSR.

Describe Node.js architecture and the event loop phases. How does it differ from the browser's loop?Senior📊 diagram

  1. timers — expired setTimeout/setInterval
  2. pending callbacks — deferred system callbacks
  3. poll — retrieve new I/O events; may block here waiting

"Node is single-threaded" — is that accurate? How do you detect and avoid blocking the event loop?Senior

  • Detect: event-loop lag metrics (perf_hooks.monitorEventLoopDelay()), APM dashboards, --prof/clinic.js flame graphs, slow p99 with idle CPU on other cores.
  • Avoid: never use *Sync APIs in request paths; chunk big loops (setImmediate between batches); stream instead of buffering; offload CPU work to worker_threads; validate/limit input sizes; bump UV_THREADPOOL_SIZE if fs/crypto-bound.

Node streams and backpressure — explain and show how to process a large file safely.Senior📊 diagram

Model answer: Streams process data in chunks with bounded memory. Four types: Readable, Writable, Duplex, Transform. Backpressure: when a fast producer outruns a slow consumer, writable.write() returns false (internal buffer ≥ highWaterMark) — the producer must pause until 'drain'. pipe()/pipeline() handle this automatically; pipeline() additionally propagates errors and destroys all streams (pipe leaks on error — always prefer pipeline).

const { pipeline } = require('stream/promises');
await pipeline(
  fs.createReadStream('huge.csv'),          // ~64KB chunks
  zlib.createGzip(),                         // Transform
  fs.createWriteStream('huge.csv.gz')        // backpressure handled
);
Bounded-memory pipeline: the slow end signals the fast end to pause.

Real uses: HTTP responses are streams (pipe a DB export to the client), CSV/report generation, proxying uploads to S3 — constant memory regardless of file size.

Stream xử lý theo chunk, RAM cố định. Backpressure: write() trả false → dừng đến khi 'drain'. Luôn dùng pipeline() thay pipe() (tự xử lý lỗi + destroy).

How do you find a memory leak in a production Node service?Senior

Model answer — a repeatable workflow:

  1. Confirm: plot process.memoryUsage().heapUsed / container RSS over time — steady climb across GCs = leak (sawtooth that returns to baseline = normal).
  2. Capture: take 2–3 heap snapshots spaced apart under load (--inspect + Chrome DevTools, or v8.writeHeapSnapshot() / heapdump in prod).
  3. Compare: DevTools "Comparison" view — sort by retained size delta; look at what's growing (arrays, Maps, closures) and follow retainer paths to find who holds the reference.
  4. Usual suspects: module-level caches without eviction, event listeners added per-request on shared emitters (watch for MaxListenersExceededWarning), closures in long-lived timers, unbounded in-memory sessions/metrics, sockets never closed.
  5. Fix & verify: LRU/TTL caches, WeakMap for object-keyed metadata, cleanup handlers; re-run load test and watch the curve flatten.

Mention allocation sampling/clinic.js heapprofiler for hard cases, and --max-old-space-size only as mitigation, never a fix.

Quy trình: xác nhận qua metric heap tăng đều → chụp 2–3 heap snapshot → so sánh, lần theo retainer path → nghi phạm quen thuộc: cache không giới hạn, listener chồng chất, timer giữ closure → fix bằng LRU/WeakMap rồi load test lại.

How does a B-tree index work? When is an index NOT used, and what is the leftmost-prefix rule?Senior📊 diagram

Model answer: A B+tree keeps sorted keys in a shallow, wide tree (hundreds of keys per node ≈ 3–4 levels for millions of rows), so a lookup is O(log n) with very few disk pages read. Leaf nodes hold pointers to rows (or the row itself in a clustered index) and are linked — which is why range scans and ORDER BY on the index are cheap.

B+tree: shallow fan-out for point lookups; linked leaves for ranges.

Index is skipped when: a function/cast wraps the column (WHERE YEAR(created_at)=2026 → rewrite as range), leading wildcard LIKE '%x', low selectivity (optimizer prefers full scan), type mismatch, OR across different columns, or stale statistics.

Composite index (a, b, c) — leftmost prefix: usable for a, a,b, a,b,c, and a + range on b (but the range stops further use of c). Not usable for b or c alone. Order columns: equality filters first, then range/sort columns. Also mention covering indexes (index-only scans) and that every index taxes writes.

B+tree nông và rộng → tìm điểm O(log n), leaf nối nhau → range scan nhanh. Index bị bỏ qua khi bọc hàm quanh cột, LIKE '%…', selectivity thấp. Composite (a,b,c): phải dùng từ trái sang; equality trước, range sau. Index nhiều = ghi chậm.

SQL vs NoSQL — how do you choose for a given feature?Senior

Model answer — decide by data shape, consistency needs, and access patterns, not fashion:

  • Relational (Postgres/MySQL): default choice. Relationships + joins, transactions across entities (payments, orders), ad-hoc queries, strong consistency. Postgres JSONB covers most "flexible schema" needs anyway.
  • Document (MongoDB): self-contained aggregates read/written as a unit, heterogeneous schemas per record.
  • Key-value/cache (Redis): sessions, rate limit counters, leaderboards, pub/sub, hot caching.
  • Wide-column (Cassandra) / time-series (Timescale, ClickHouse): huge write volumes, known query patterns — e.g. market tick/candle data for charting.
  • Search (Elasticsearch): full-text, faceting.

Mặc định chọn SQL (Postgres) — join, transaction, truy vấn linh hoạt. NoSQL khi: aggregate độc lập (Mongo), cache/counter (Redis), ghi cực lớn theo pattern biết trước (Cassandra/time-series cho tick data), full-text (ES). Thực tế thường kết hợp.

What is the N+1 query problem and how do you fix it?Mid+

Offset vs cursor (keyset) pagination — why does offset break at scale?Senior📊 diagram
Offset scans everything it skips; keyset seeks directly via the index.

OFFSET đọc-bỏ n dòng → càng sâu càng chậm, trang bị lệch khi có insert. Cursor/keyset: WHERE (ts,id) < cursor LIMIT 20 — seek theo index, chi phí không đổi. Load nến cũ của chart chính là keyset pagination.

Replication vs sharding: how do you scale a database, and what is replication lag?Senior📊 diagram
Replicate for reads and failover; shard when a single primary can't hold the writes or the data.

Model answer: Scale in order: (1) optimize queries/indexes, (2) cache (Redis), (3) read replicas — async copies serving reads, plus failover; (4) sharding — horizontal partitioning by a shard key, last resort because cross-shard joins/transactions and resharding are painful.

Replication lag = replica is seconds behind primary → user writes then reads a replica and sees stale data. Shard key criteria: even distribution (avoid hot spots — e.g. celebrity users), aligns with query patterns to keep queries single-shard.

Thứ tự scale: index/query → cache → read replica → shard (cuối cùng vì join/txn xuyên shard rất khổ). Lag: ghi xong đọc replica thấy data cũ → route đọc-sau-ghi về primary. Shard key phải phân bố đều + khớp query pattern.

Why is connection pooling critical in Node? How do you size the pool?Senior

  • Node specifics: one process handles thousands of concurrent requests over a pool of ~10 connections — requests queue for a free connection. Under-sized pool → queue timeouts under load; oversized → you DDoS your own DB (Postgres max_connections is small, ~100 default).
  • Sizing: total = pool_size × process_count × service_instances ≤ DB max. A good starting heuristic: connections ≈ (cores × 2) + effective_spindle_count on the DB side; measure pool wait time and tune. Use PgBouncer when many instances share one DB.
  • Pitfalls: leaking connections by not releasing on error paths (use finally), long transactions hogging the pool, running migrations through the pooled app user.

Mở connection mỗi request = handshake + process DB mới → chết latency. Pool ~10 conn phục vụ ngàn request (xếp hàng). Tổng conn mọi instance phải ≤ max của DB; nhiều instance → PgBouncer. Luôn release trong finally.

Containers vs VMs — what actually isolates a Docker container under the hood?Senior📊 diagram

Model answer: A container is not a lightweight VM — it's an ordinary Linux process wearing isolation glasses. Three kernel features do all the work:

  • Namespaces — what the process can see: its own PID tree (PID ns), network stack/interfaces (net ns), filesystem root (mount ns), hostname (UTS), users (user ns), IPC.
  • cgroups — what it can use: CPU, memory, I/O limits (this is what --memory=512m sets; exceed it → OOM-killed).
  • Union/overlay filesystem — image layers stacked read-only + one writable layer on top; layers are shared between containers, which is why 10 containers from one image cost almost nothing extra on disk.
VMs virtualize hardware and boot a full OS each; containers share the host kernel — start in ms, cost MBs.

Consequences worth stating: containers share the host kernel → weaker isolation boundary than a VM (why multi-tenant platforms run containers inside VMs); a container "boots" in milliseconds because nothing boots — it's just fork+exec with namespaces; Docker on Mac/Windows actually runs a hidden Linux VM.

Container = process Linux bình thường + namespaces (thấy gì) + cgroups (dùng bao nhiêu) + overlay FS (layer chia sẻ). Chung kernel với host → nhẹ, khởi động ms, nhưng cách ly yếu hơn VM. Docker trên Mac chạy trong 1 VM Linux ẩn.

How do you deploy frontend assets to a CDN safely? (versioning, invalidation, rollback)Senior · CV

Model answer: The core trick is immutable, content-hashed assets:

  1. Build produces app.<hash>.js — upload to S3 first, with Cache-Control: public, max-age=31536000, immutable. New files never conflict with old ones.
  2. Deploy the entry point (HTML or a manifest) last, with no-cache (revalidate every time) — flipping it atomically switches all users to the new asset graph.
  3. CloudFront invalidation only for the entry point (/index.html) — cheap and instant; hashed assets never need invalidation.
  4. Rollback = re-point the entry file to the previous hashes (keep N old builds in the bucket) — seconds, no rebuild.

Edge cases that show experience: users with an open SPA requesting old chunks after deploy (keep old assets available + handle chunk-load errors by prompting refresh); SDK-on-CDN consumers pinned to versioned URLs (/sdk/1.4.2/) plus a controlled latest alias; monitor error rates post-deploy before deleting anything.

Asset có hash + immutable 1 năm, upload trước; HTML/manifest no-cache deploy sau cùng (chuyển version nguyên tử). Chỉ invalidate entry point. Rollback = trỏ lại hash cũ (giữ N bản build). SPA đang mở có thể request chunk cũ → giữ asset cũ + xử lý lỗi load chunk.
